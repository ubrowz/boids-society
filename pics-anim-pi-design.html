Overview

An animated browser-based population simulation built on the classic boids algorithm. Coloured boids wander autonomously inside a bounded aquarium. They can bond, breed, age and die. A multi-arm feedback loop attempts to maintain a stable population around a user-defined target.

Since July 2026 the file opens as a control dashboard: the simulation itself runs fullscreen in a separate tab (the aquarium), launched from the dashboard and remote-controlled live over a postMessage bridge — see Dashboard & Run Modes.

The rendering engine is PixiJS v7 (WebGL), replacing the original Canvas 2D path. The PixiJS source (v7.4.2, ~450 KB) is embedded inline in the HTML — no CDN fetch at load time — so the file and every export from it run fully offline (essential for a Pi without internet access). Boid and predator bodies are drawn as textured sprites (teardrop-v3.png, embedded as base64); trail lines, tail fins, beaks, and effects are drawn with PIXI.Graphics. This gives a large framerate gain on the Raspberry Pi 4, which benefits strongly from GPU-accelerated sprite batching.

The visual layer extends the base simulation with:

  • Motion trails — single-pass core line behind every moving boid and predator
  • Forked tail fin — a V-shaped Graphics polygon drawn behind each boid body, peeking out from behind the sprite
  • Birth rings and death splashes — expanding ring on every birth; multi-ring particle burst on every death (predator kill, natural death, disease)
  • Morphological genetics — males carry a beakGene (beak length), all boids carry an eyeGene (eye size), both heritable and mutable via bit-flip
  • Colour as HSL — boids store {hue, sat}; rendered via sprite tint at a user-adjustable lightness (50–100%, default 85%)
  • Colour family cycle — a preferred family (red → yellow → green → blue → magenta → …) cycles automatically; three selection pressures drive convergence
  • Predators — autonomous hunters that preferentially eat wrong-family boids; rendered as white teardrop sprites identical in shape to boids
  • Details toggle — hides beak, forked tail, eyes, aura glow, disease dot and motion trails with one button; birth rings and death splashes always show
  • Colour toggle — switches all boid rendering to greyscale tones; colour wheels, rings and splashes are unaffected
  • Depth (2.5D) — an optional user-defined depth range: boids drift in z, near boids render larger than far ones, and hard collisions become 3D (see Depth)
  • Upright toggle — pins the body sprite to its original orientation instead of rotating with the movement direction (for photo bodies)
  • Export button — generates a standalone full-screen HTML with no UI, auto-start, auto-disease

Target framerate is 30 fps (capped; simulation time is dt-based so it runs correctly at any framerate). All logic runs in a single self-contained HTML file with no external dependencies.

Boid Appearance

Both sexes — body sprite

Rendered as a PixiJS Sprite using the embedded PNG texture teardrop-v3.png (80×80 px RGBA, anchor at pixel 40,40). The sprite is scaled to RADIUS / 13 and tinted to the boid's colour via PixiJS's multiplicative tint. The PNG encodes shape and greyscale shading; the tint supplies hue. Nose (forward) is at pixel 72 (+32 px from anchor → 2.46r ahead in world space); tail base is at pixel 5 (−35 px → 2.69r behind).

Tinting: PixiJS tint is multiplicative — each pixel channel is multiplied by the tint colour. A white pixel in the PNG becomes exactly the tint colour; a greyscale mid-tone is darkened proportionally. To keep bright highlights visible the tint lightness is user-adjustable via the tint L slider (50–100%, default 85%): tintHex = hsl(hue, sat×100%, SPRITE_L). In greyscale mode (colour toggle off), the tint is set to 0xffffff (no colour shift) so the PNG's own grey shading is preserved at full brightness.

Both sexes — forked tail fin

A V-shaped forked tail is drawn behind each boid as a PIXI.Graphics polygon in the tails layer (below the sprite layer), so it visually peeks out from behind the body. The fill colour is the boid's hue at lightness 68% (lighter than the 50% body) — except when the boid is the locked target of a predator in lock hunt mode, in which case the tail fin is drawn white as a targeting indicator. Key geometry constants:

junction   = 1.70r back  (where the fork starts, inside sprite tail)
notch      = 2.70r back  (centre indent between the two lobes)
tip        = 3.00r back  (furthest point of each lobe)
spread     = ±0.55r sideways at the tips

Males only — beak

A forward-pointing tapered beak projects from the sprite nose edge at noseEdge × 0.93. The tip reaches noseEdge + beakLen ahead of the boid centre. Beak length is determined by beakGene: RADIUS × (0.3 + beakGene/255 × 1.7), giving a range of roughly 0.3r (stub) to 2.0r (long spike). Females carry beakGene silently but do not express it.

Rotation smoothing

Each boid maintains a separate _drawAngle field that lerps toward the physics heading at factor 0.20 per frame. All drawing (sprite rotation, tail fin, beak, eyes) uses _drawAngle, not the raw velocity angle. On wall bounces, _drawAngle is snapped immediately to the new heading to avoid lerping through a 180° arc.

Movement

Each boid moves at a constant speed, direction updated every frame by four competing steering forces applied in sequence:

  1. Wall avoidance — soft repulsion begins 40 px from any wall; hard bounce at the wall itself
  2. Wander — a random-walk steering target projected ahead of the boid on a circle, jittered each frame
  3. Nearness — attraction toward the local flock centre (see Feedback Arm 1)
  4. Bond attraction — bonded partners stay within 45 px of each other

Speed is user-controlled via a slider (1–10, mapped to 0.45–4.5 px/frame).

Depth (2.5D)

The depth slider (0–600, default 0 = off) gives the aquarium a third dimension without abandoning the 2D architecture. Each boid carries a z coordinate in [0, DEPTH] (0 = nearest to the viewer) and a z-velocity vz. At depth 0 every dz is zero and every scale factor is 1, so the simulation is bit-for-bit the original 2D behaviour.

z-motion

The xy steering model (heading angle + constant speed) is untouched. Depth motion is an independent 1D random walk per boid (updateZ): vz is jittered each frame, capped at 35% of the planar speed so depth reads as drift, with soft repulsion inside a 60 px band at the near/far walls and a hard clamp at the boundaries. Bonded boids additionally converge gently toward their partner's z so pairs can actually meet in 3D. New boids spawn at a random depth; children are born at their parents' mean depth.

Perspective rendering

The legacy 2D plane sits at mid-depth: scale = Z_FOCAL / (Z_FOCAL + z − DEPTH/2) with Z_FOCAL = 600. A boid at z = DEPTH/2 renders at exactly legacy size; nearer boids grow beyond it (up to 2× at z=0 with depth 600) and farther ones shrink (down to 0.67×). Because the body sprite, tail fin, beak, eyes and glow all derive their geometry from the boid radius r, scaling r by this factor scales everything coherently. Trails and bond lines thin with distance, death splashes and birth rings capture the depth scale at spawn time, and boids beyond the midplane dim slightly (alpha × min(1, 0.45 + 0.55·scale)). Draw order is painter's, applied in every mode: boids and predators share one body-sprite pool and are drawn in a single depth-sorted pass, sorted far-first on continuous z with a descending-id tie-break (predators carry negative ids, so they never collide with boid ids). This makes them occlude each other correctly — a near predator covers far boids; a far predator sits behind near ones — and because the sort key is the exact depth, draw order always matches on-screen size: a nearer (bigger) entity can never be drawn behind a farther (smaller) one. Front/back flicker between entities at nearly equal z is prevented not by the sort but by the depth-layering force (see “Volume & depth layering”), which pushes on-screen-overlapping boids apart in z so their z-values stop crossing. The id tie-break gives a stable order at depth 0, where all z = 0 — newborns fade in behind older boids instead of rendering on top of everything (~60 µs/frame on a Pi 4 at 200 entities). (An earlier version quantised z into 16-unit bands with an id tie-break within each band; that was dropped because the within-band id ordering ignored true depth, letting a smaller boid draw in front of a bigger one.) The small above-body overlays (eyes, beak, glow, the predator's beak/eyes) still draw in grouped layers over all bodies, so a far entity's overlay features can poke over a nearer body — the bodies themselves interleave correctly.

Simulation semantics

Which interactions read depth is a deliberate split. Bonding hits stay 2D (xy only) — they are part of the population feedback loop tuned in 2D, and (crucially) the depth-layering force pushes on-screen-overlapping boids apart in z, so a 3D bonding test would hold courting pairs permanently out of range and no bond could form. Bonding therefore fires on on-screen encounter, exactly as it did before depth existed. Disease transmission, predator eating and body-separation (in-plane push) use 3D distance (dx² + dy² + dz²), so boids at different depths pass over each other without physical contact. The rest of the steering — nearness/cohesion, pressure sensing, wall avoidance — is also 2D, so the population feedback loops never need retuning when depth changes. The 2D spatial grid is the broad phase throughout; any z check happens in the narrow-phase distance test.

Volume & depth layering (intrinsic in 3D)

When depth is enabled, boids occupy volume and do not interpenetrate — separateBoid runs every frame regardless of the No overlap toggle (which then only governs flat 2D mode). Two independent position-projection forces:

  • In-plane push — gated on true 3D proximity (d² < (2·RADIUS)²), so a boid far in front never shoves one far behind out of the plane. Position nudge (8% of overlap) plus a steerToward impulse (strength 5) so headings navigate around each other.
  • Depth layering — boids that overlap on screen repel each other in z toward distinct depths, strongest when nearly co-planar and scaled by visual overlap; exact-z ties broken by id. With the continuous depth sort this is the jitter damper: two boids sharing screen space settle at slightly different depths so their z-values stop crossing frame-to-frame, killing the flicker. (Order/size consistency is guaranteed by the continuous sort itself; layering only stops the crossings.)
    Reach is each boid's true rendered footprint, not a constant. On-screen sprite size is native image px × (RADIUS · depthScale / bodyScale) — the native image pixel size is not normalised, so pool images of different pixel dimensions render at different sizes (by design: load a smaller image for a smaller boid). Each boid therefore records its own on-screen half-extent _visR = ½·max(texW,texH)·scale in drawBoid, and the layering treats two boids as overlapping when their centre distance is below _visR(a) + _visR(b). A fixed trigger would miss large-image boids and leave them jittering. The neighbour query widens to _visR + _maxVisR (largest footprint this frame) so no overlap is missed. The in-plane non-interpenetration push keeps its fixed 2·RADIUS body reach (physics bodies are image-independent). Depth gap is deliberately small: _SEP_Z = 1.5·RADIUS ≈ 30 units, capped at DEPTH. Because the sort is continuous, the gap doesn't need to clear any quantum — it only has to exceed the per-frame z random-walk step so the gap's sign holds (push coefficient 0.12 > the jitter step).

No exemptions are needed. Bonded partners rest ~one body apart in the plane (they don't overlap on screen), so they receive almost no z-push and stay near-coplanar with a short bond line. And because the collision that repels is the very event that banks a bonding hit — and hits never decay — repulsion only paces courtship (bump → drift apart → re-approach → bump), it can never prevent bonding; at BOND_THRESHOLD = 1 the first contact bonds regardless. The quantized z-band sort remains as the backstop for the rare genuine crossing inside a dense knot.

Known trade-offs (v1)

  • Encounter dilution — depth dilutes density, so bonding (and disease spread) is measurably slower at high depth: the volume grows but interaction radii stay constant. At depth 600 expect a visibly slower ecosystem; depth 200–400 is a good balance.
  • Overlay features draw above all bodies — eyes, beak, glow and the predator's beak/eyes live in Graphics layers over the shared body container, so a far entity's small overlay features can show over a nearer body. Bodies themselves interleave correctly by depth.
  • Layer artefacts — tail fins and predator fins live in a Graphics layer below all bodies, so a near entity's fin can briefly hide behind a farther body during screen overlaps.

Singles

Each boid is assigned a "single" status at birth, drawn probabilistically from the singles slider (0–80%, default 20%). Singles never bond. They are drawn at 0.7× the normal radius to visually distinguish them.

Sexual Orientation

Each non-single boid is assigned an orientation at birth:

  • Straight (default) — can only bond with an opposite-sex non-gay boid
  • Gay — can only bond with a same-sex gay boid

The gay % slider (0–50%, default 10%) sets the fraction of non-single boids born gay. Singles take precedence: a boid that is single is never also marked gay. Gay pairs bond and divorce normally but can never breed. The feedback loop's deadlock detection and immigration system ignore gay boids.

Immigrants are always straight, as they are spawned specifically to break reproductive deadlocks.

Bonding

Two unbonded, non-single boids bond after accumulating BOND_THRESHOLD collisions (overlapping circles). A per-pair cooldown prevents one slow overlap from counting as many hits.

Restrictions:

  • A bonded boid cannot bond again until its partner dies or the pair divorces
  • A child cannot bond with either parent
  • Siblings (same two parents) cannot bond with each other
  • Singles never bond
  • Gay boids only bond with gay boids of the same sex; straight boids only bond with straight boids of the opposite sex

Bond threshold interaction with nearness: Higher BOND_THRESHOLD requires more collisions, so boids need sustained proximity. The nearness strength is scaled upward proportionally (1 + (threshold - 1) × 0.24) to compensate. The hit cooldown also shrinks with higher thresholds (COOLDOWN_FRAMES / √threshold) so hits accumulate faster.

Morphological bonding preferences

For straight M+F collision pairs, each hit probability is further scaled by the morphological attractiveness of the two individuals:

eyeMult  = 0.75 + female.eyeGene/255 × 0.5   # 0.75..1.25, eyes alone
beakMult = 0.75 + male.beakGene/255  × 0.5   # 0.75..1.25, beak alone
hitProb  = min(1, hitProb × eyeMult × beakMult)   # combined 0.56..1.56
  • Each trait is selected independently. Big eyes always help a female bond, regardless of the male's beak length — and vice versa.
  • At average genes (128/255 ≈ 0.5), both multipliers are 1.0 → no change to base behaviour
  • One trait maxed, other average → multiplier = 1.25 (25% more likely)
  • Both traits maxed → multiplier = 1.56 (56% more likely)
  • Both traits minimal → multiplier = 0.56 (44% less likely)

This creates independent runaway sexual selection on both traits: big-eyed females and long-beaked males each bond more readily on their own merits, driving both traits upward over generations. The preference applies only to straight pairs; gay pairs are unaffected.

Divorce

A pair splits on either of two triggers: the random DIVORCE_RATE × dt chance each frame, or the distance limit below (they "grew apart"). When a divorce occurs:

  • Both partners become unbonded and their hit/children records for that pair are cleared
  • Each partner's breed timer is reset with a random offset
  • pairFamilyTarget is reset to 0 for both

The divorce rate slider (0–50%, default 5%) sets the per-second probability.

Bond distance limits & reconvergence ramp

The bond is a soft spring (steer toward partner beyond BOND_DIST = 90 px), not a rigid leash, so partners can be pulled apart by competing forces. Two distance limits give that drift consequences, and a turn-rate ramp keeps most couples from ever reaching them:

  • Limit A — no offspring (BOND_BREED_DIST = 2·BOND_DIST = 180 px). A couple farther apart than this cannot conceive: updateBreeding bails on a squared-distance check (no sqrt) before decrementing the breed timer, so conception simply pauses while they are separated and resumes when they close up.
  • Limit B — divorce (BOND_DIVORCE_DIST = 3.5·BOND_DIST = 315 px). Beyond this the pair divorces ("grew apart"), via the same split path as a random divorce. Checked in updateDivorce with a squared distance.
  • Reconvergence ramp (BOND_TURN_RAMP = 4). Raising BOND_STRENGTH alone does nothing — the update loop's smooth-angle cap (TURN_SPEED×1.5) overrides it, so a boid heading away turns back only ~4°/frame, which is why couples drift. Instead, bondAttraction lifts that cap with distance: _turnBoost = 1 + urg·BOND_TURN_RAMP where urg ramps 0→1 from BOND_DIST to BOND_DIVORCE_DIST. A far-strayed boid U-turns up to 5× faster and reels back before hitting limit B — so ordinary wandering apart does not automatically end the marriage; only a sustained, strong pull does.

All three are constants derived from BOND_DIST (not sliders), so they need no bridging or export baking. Cost is negligible on the Pi: the breeding check adds one squared-distance test per fertile pair; updateDivorce adds one per pair but now skips its per-frame Set allocation (using an id<partner.id guard), so it is net neutral-to-cheaper; the ramp is one multiply per boid.

Sex and Fertility

Each boid is assigned a sex (M or F) at birth, biased toward whichever sex is currently underrepresented to keep the ratio close to 50/50.

Fertility windows are fixed fractions of each boid's maxLifespan:

ParameterValue
♂ fertile from20% of maxLifespan
♂ fertile for70% of maxLifespan
♀ fertile from20% of maxLifespan
♀ fertile for50% of maxLifespan

A boid is fertile only when age falls within [maxLifespan × from, maxLifespan × (from + for)]. Using maxLifespan (not the pressure-varying effectiveLifespan) means each boid's reproductive window is a fixed fraction of its own biology and does not shift as population pressure changes.

Only bonded straight M+F pairs where both partners are currently fertile can produce children.

Breeding

When a bonded fertile M+F pair is active, a per-pair family target determines reproductive capacity. At bond formation the pair's family target is drawn from Normal(FAMILY_SIZE, 0.8), clamped to a minimum of 0 and rounded to the nearest integer. This is stored on both partners as pairFamilyTarget.

The effective maximum children is then:

t              = (globalPressure + 1) / 2          # 0..1
desiredMax     = ceil(pairFamilyTarget × (1 − max(t − 0.5, 0) × 2))
effectiveMax   = min(desiredMax, bioCeiling)
bioCeiling     = floor(maxLifespan × F_FERTILE_FOR / PREG_INTERVAL)
  • At target (t = 0.5): desiredMax = pairFamilyTarget (no reduction)
  • Overpopulated (t = 1): desiredMax = 0 (breeding stops)
  • At crisis (t = 0): desiredMax = pairFamilyTarget (no bonus above the family target)

The actual pregnancy interval is modulated by pressure:

intervalScale  = 3^globalPressure
actualInterval = PREG_INTERVAL × intervalScale × random(0.8, 1.6)

The Feedback Loop

Stability is maintained through three independent feedback arms, all driven by pressure signals derived from the current population state.

Pressure Signals

Local pressure (localPressure, 0–1) — measures neighbourhood crowding:

localPressure = min(neighbours / max(TARGET_POP × 0.4, 1), 1)

The sensing radius (SENSING_RADIUS) is user-adjustable via the sensing radius slider (30–300 px, default 130 px). It controls how far each boid can "see" for pressure calculation, neighbour attraction, and collision detection.

Global pressure (globalPressure, −1 to +1) — measures population-level error:

targetPairs    = (TARGET_POP / 2) × min(M_FERTILE_FOR, F_FERTILE_FOR)
fertilePairs   = min(fertile males, fertile females)
globalPressure = clamp(fertilePairs / targetPairs − 1, −1, +1)

An extinction floor overrides globalPressure to −1 when total population drops below 4.

Population pressure (popPressure, −1 to +1) — measures raw headcount error (Arm 1 only):

popPressure = clamp(boids.length / TARGET_POP − 1, −1, +1)

Arm 1 — Nearness (birth rate, indirect)

Signals: localPressure and popPressure — controls how strongly boids steer toward each other. Cohesion is computed from neighbours within BOND_DIST (90 px), not the full SENSING_RADIUS, keeping flocking attraction local and preventing distant groups from pulling each other across the arena.

combinedPressure = min(localPressure + max(popPressure, 0), 1)
strength = (MIN_NEARNESS + (BASE_NEARNESS − MIN_NEARNESS) × (1 − combinedPressure))
           × thresholdScale

Why three separate signals?

Early versions used a single blended pressure signal for all arms. This caused a critical failure: when population was low and nearness caused boids to cluster, local density increased, which raised pressure, which suppressed recovery — the system sabotaged itself. Separating the signals prevents this: Arm 1 responds to local density + raw headcount overshoot; Arms 2 and 3 respond only to global population-vs-target error.

Genetic System

Colour as genome

Each boid carries {hue: 0–360, sat: 0–1} as its genome. Hue encodes colour family membership; saturation encodes vividness. Children inherit colour through inheritColor (see Colour Family Cycle). beakGene and eyeGene continue to use bit-flip mutation (see Morphological Genetics).

Genetic distance preference

When two unbonded boids collide, the cooldown before the next registered hit is modulated by their genetic distance:

let dh = |a.hue − b.hue|
if dh > 180: dh = 360 − dh
geneticDistance = dh / 180        # circular hue distance, 0..1
cooldownScale   = 2.0 − 1.5 × geneticDistance

Boids with similar hues (distance ≈ 0) have 2× longer cooldowns — they bond slowly. Hue-distant boids (distance ≈ 1) have 0.5× cooldowns — they bond up to 4× faster. This continues to favour outbreeding.

Inbreeding depression

As diversity falls, two independent fitness penalties activate:

t = 1 − clamp((diversity − 0.05) / (0.3 − 0.05), 0, 1)
  • Mortality penalty (inbreedingHazardMultiplier = 1 + t × 0.5, up to 1.5×) — models general fitness reduction from inbreeding.
  • Disease susceptibility penalty (diversityFatalityMultiplier = 1 + t × 2.0, up to 3×) — models HLA diversity collapse. With 10% base fatality, full convergence pushes effective fatality to 30%.

50/500 rule

Effective breeders = fertile bonded M+F pairs × 2.

smallPopHazardMultiplier = 1 + (1 − min(effectiveBreeders, 50) / 50) × 0.2

The 50/500 rule from conservation biology: < 50 effective breeders = immediate viability risk; < 500 = long-term drift risk. At typical simulation scales (target 30), effective breeders are almost always below 50.

Immigration

When immigration is enabled, an immigrant boid may be spawned based on combined deficit pressure:

popDeficit     = max(1 − boids.length / TARGET_POP, 0)
fertileDeficit = max(−smoothGlobalPressure, 0)
pressure       = max(popDeficit, fertileDeficit)

Immigration also triggers on genetic deadlock: when no valid unrelated straight M+F fertile non-single pairs exist. The spawn interval shortens as pressure increases (minimum ~5 s at full crisis). Immigration is suppressed entirely when population exceeds 120% of target.

Immigrants arrive with a fresh random colour and genome, at mid-fertile-age, assigned the sex most needed for balance. Never singles, always straight. Immigrants fade in over BIRTH_DURATION (1.5 s) exactly like naturally-born boids, and emit a white birth ring on arrival (natural births use the boid's own hue colour, so white clearly signals an immigration event).

Disease

A contagious disease can be introduced by pressing infect. In exported files, disease occurs automatically on a random 30–90 s timer (AUTO_DISEASE).

Transmission

Healthy boids accumulate exposure hits whenever they physically overlap with an infected boid. A per-pair cooldown (DISEASE_COOLDOWN = 1.5 s) prevents rapid accumulation from a single sustained contact. Once hits reach DISEASE_THRESHOLD, the boid becomes infected.

Fatality

Each frame while diseased, a boid faces an additional death roll independent of the Gompertz hazard:

effFatality = min(0.9999, DISEASE_FATALITY × diversityFatalityMultiplier)
h_disease   = −ln(1 − effFatality) / DISEASE_DURATION
P(die from disease this frame) = h_disease × dt

Deaths are spread naturally across the illness period. effFatality rises up to 3× base with diversity loss.

Recovery, immunity and R-factor

Infected boids that survive the full DISEASE_DURATION recover and become permanently immune. The R-factor is the rolling mean of infectCount over the last 30 recovered or deceased infected boids.

Motion Trails

Every boid and predator stores a fixed-length ring buffer of past positions (trail[]). Trails are rendered in a single core pass with t = i / (n−1) (0 = oldest, 1 = newest):

Birth Pulse

When a child is born a short expanding ring pulses outward from the newborn itself, reusing the existing birthTimer field (no extra data structure). As t = birthTimer/BIRTH_DURATION counts down from 1 to 0:

radius    = RADIUS × (1 + (1−t) × 4)   # expands outward
lineWidth = t × 8                        # thins as it expands
alpha     = t² × 0.9                     # fades out quickly

Colour matches the newborn boid's own colour. The ring is drawn in the same drawBoid pass — no separate array or draw function.

Death Splashes

Every boid death — whether by predator kill, natural aging (Gompertz), or disease — spawns a deathSplash at the boid's position, rendering over ~0.85 s. All colours are derived from the dead boid's own hue and saturation — a blue boid produces a blue burst, a red boid a red burst, etc. The splash is identical regardless of cause of death.

Morphological Genetics

Both beakGene and eyeGene are 8-bit integers (0–255). They are heritable, blended at birth, and mutated via bit-flip:

childGene = mutateGene( round( (parentA.gene + parentB.gene) / 2 ) )

# With probability MUTATION_RATE, one random bit is flipped:
childGene = parentGene XOR (1 << randomBit)   [masked to 8 bits]

Low bits produce tiny phenotypic shifts; flipping bit 7 (MSB) produces a dramatic jump of ±128 gene units.

Colour Representation & Mutation

Each boid stores colour as {hue: 0–360, sat: 0–1} and is rendered as hsl(hue, sat×100%, 50%). Lightness is fixed at 50% so all boids are equally vivid regardless of hue. sRGB is used throughout (Canvas 2D default).

Colour families

Hue maps to six named families used by the cycle system and predator preference:

FamilyHue rangeCSS colour

Boids with sat < 0.15 are treated as colourless (gray) and count as wrong-family for all selection pressures.

Mutation

With probability MUTATION_RATE, a small random jitter is applied to the child's hue and saturation after inheritance:

childHue = (hue + uniform(−15°, +15°) + 360) % 360
childSat = clamp(sat + uniform(−0.1, +0.1), 0.2, 1.0)

Colour Family Cycle

The simulation maintains a preferred colour family that the population is driven toward through three independent selection pressures. Once ≥ 90% of living boids belong to the preferred family for 5 consecutive seconds the target advances to the next family in the cycle:

red → yellow → green → cyan → blue → magenta → red → …

The current family, convergence percentage and countdown to advance are shown in the status panel.

Pressure 1 — bonding (family pull slider)

Each potential bonding hit is gated by a matchScore based on how many of the pair are in the preferred family:

matchScore = (aInFamily ? 0.5 : 0) + (bInFamily ? 0.5 : 0)   # 0, 0.5, or 1
hitProb    = 1 − (1 − matchScore) × FAMILY_PULL × 0.85
if random() > hitProb: skip this hit
SituationmatchScorehitProb at 60% pull
Both in preferred family1.0100%
One in preferred family0.5~75%
Neither in preferred family0.0~49%

At 0% pull there is no preference. At 100% pull wrong-family pairs have only a 15% hit chance, bonding roughly 7× slower than preferred-family pairs.

Pressure 2 — predator preference

Predators preferentially hunt and eat boids that do not belong to the preferred family. When veryHungry (no meal for 2× FULL_DURATION) they eat any boid regardless of family. See Predators below.

Pressure 3 — biased inheritance

Every birth nudges the child's hue toward the preferred family's representative hue, regardless of parent colours:

function mixHue(h1, h2):
    d = h2 − h1
    if |d| > 180: d -= sign(d) × 360
    return (h1 + d × 0.5 + 360) % 360

function biasedHue(h1, h2, preferredHue):
    child = mixHue(h1, h2)
    d = preferredHue − child
    if |d| > 180: d -= sign(d) × 360
    return (child + 0.3 × d + 360) % 360

The factor 0.3 is the evolutionary pressure constant — every child moves 30% of the remaining hue distance toward the target. When at least one parent is already in the preferred family, a stronger dominant inheritance rule applies: the child's hue is snapped directly into the preferred family, using the preferred parent's hue with a small ±10° jitter.

Predators

Predators are autonomous agents separate from the boid population. They do not flock, do not age, and do not breed.

Appearance

Predators use the same teardrop PNG sprite (teardrop-v3.png) and forked tail fin as boids, tinted white (0xffffff). The PNG's greyscale shading is preserved by multiplicative tinting, giving a white body with natural depth. Predators are drawn at PREDATOR_RADIUS = RADIUS × 0.7 — the same size as a single boid — so they blend into the flock and are harder to spot.

  • Body sprite — teardrop-v3.png at scale PREDATOR_RADIUS / 13, tint 0xffffff, alpha 0.92
  • Forked tail fin — same V-fork geometry as boid tails, filled near-white (0xf0f4ff, 90% alpha), drawn in the predator's own predFin layer below the sprite
  • Beak — always present (predators always express a beak); light grey (0xd0d8e8), same quadratic-curve shape as male boid beaks, length PREDATOR_RADIUS × 0.9
  • Eyes — two circles at noseEdge × 0.55 forward; colour transitions yellow (full) → orange → red (very hungry); each has a small white highlight dot; size increases slightly when hungry
  • Back dot — small filled circle on the rear of the body, coloured in the current protected family's CSS colour

Predators maintain a _drawAngle with lerp factor 0.18, snapped immediately on wall bounces — same smoothing approach as boids.

Movement

Speed = SPEED × 1.2 (20% faster than boids at the same speed setting). Predators use the same angle-capped steering as boids via the _predSteer(pred, spd, tx, ty, strength) helper:

ts   = TURN_SPEED_BASE × (spd / SPEED_MAX)
turn = clamp(angleDiff, −ts×strength, +ts×strength)
pred.vx = cos(currentAngle + turn) × spd
pred.vy = sin(currentAngle + turn) × spd

This is structurally identical to steerToward() used by boids — no velocity lerp, no per-frame speed normalisation. The heading changes by at most one smooth step per frame, giving the same movement quality as a regular boid.

Two hunt modes are selectable via the hunt toggle button (HUNT_MODE):

Hunt mode: bump (default)

When hungry (time since last meal > FULL_DURATION):

  1. Scan all living boids within PREDATOR_SENSE = 200 px
  2. Target the nearest edible boid; if none in range, wander
  3. If veryHungry (> 2× FULL_DURATION), target any boid regardless of family
  4. Steer toward target via _predSteer(pred, huntSpd, target.x, target.y, 1.0)

Target is re-evaluated every frame — the predator always chases whoever is currently nearest.

Hunt mode: lock

When hungry, the predator picks a random edible boid within PREDATOR_SENSE and commits to chasing that specific boid (pred.lockedBoidId) until it is eaten. The lock is validated every frame via boidMap: if the locked boid dies from any cause (old age, disease, another predator), the lock is released and a new random target is chosen. The locked boid's tail fin turns white as a visual indicator. In lock mode, bystander boids that pass in front of the beak are ignored — only the locked target can be eaten. The lock clears when the predator becomes full, so each new hunger episode picks a fresh random victim.

When full (both modes): wander at 60% speed; lock clears. Wander uses identical geometry to the boid wander() function — a target is projected 30 px ahead along the current heading, shifted by a 20 px random-walk offset, then steered toward at strength 0.7. Wall avoidance: angle-based steer (same pattern as avoidWalls() for boids), strength 2.0, applied after wander/seek so walls always take priority. Hard velocity bounce at canvas edges snaps _drawAngle. Predators do not wrap around.

Depth (3D)

With depth on, predators move in the third dimension too. Each carries a z and vz and does the same soft-walled random z-walk as boids; when hunting, it also dives or climbs toward the target's depth (vz += (target.z − pred.z) × 0.02), so the chase is genuinely 3D. The eating test is a 3D proximity check — the predator must reach the prey's depth, not merely cross it on screen. In rendering, the predator's size, trail width and body opacity all scale with depth via zScaleOf(pred) exactly like boids, and its body sprite interleaves with the boids in the shared depth-sorted draw pass (see Depth · Perspective rendering). At depth 0 predators stay pinned to the plane — unchanged 2D behaviour.

Eating is detected when the beak tip enters the boid's body radius. The beak tip position is computed each frame from the physics heading:

noseEdge = PREDATOR_RADIUS × (32/13)   # sprite nose geometry

This matches the visual moment the beak point touches the boid body. Eating kills the boid immediately, spawns a death splash, and resets the predator's lastEaten timer.

Starvation

If there are no living boids at all, the predator's noPreyTimer increments. After STARVATION_TIME = 45 s of zero available prey, the predator dies and is removed.

Count management

The predators slider (0–5) is reactive: increasing it spawns new predators immediately; decreasing it removes the excess. On simulation restart, predators are respawned according to the current slider value.

Dashboard & Run Modes

The file is a control dashboard when opened directly; the simulation runs in a separate tab. A RUN_MODE marker (injected into the <head> of generated documents) selects one of three modes:

  • dash (default) — the dark control-room UI: grouped control panels, live status, pyramid. No PixiJS/WebGL context is created; the canvas stays hidden. Suitable for hosting as a public website (single self-contained file, works from any static host).
  • tank — created by the open aquarium button: buildStandaloneHTML('tank') serializes the live DOM (all current settings baked in), opens a new tab via window.open + document.write, and auto-starts fullscreen.
  • standalone — the classic ⬇ export for the Pi kiosk: identical to tank but with auto-disease enabled and no bridge.

The dashboard ⇄ tank bridge

Dashboard and tank talk over postMessage (works on file:// and https alike):

  • Controls → tank: every slider/colour/number input and each toggle-button click is forwarded as a {t:'ctl'} message. The tank applies it by driving its own hidden copy of the control (set value + dispatch input, or click()) so every wire() handler and side effect is reused verbatim — no duplicated setter logic. A custom body PNG is forwarded as a data URL.
  • Tank → dashboard: ~3× per second the tank mirrors its status DOM (population, rates, gene-pool stats, colour target, bars) as a compact snapshot, and ships the population pyramid as a PNG data URL at the pyramid refresh cadence. The header shows a green link dot and “aquarium live · N boids”.
  • The ▶ button remote-controls the tank (restart/pause); with no tank open it opens one. Closing the tank tab leaves the dashboard in its “no aquarium open” state; the button relaunches with current settings. Closing the dashboard leaves the tank swimming autonomously with its last settings.

Export

The ⬇ export button generates a standalone boids-aquarium.html file:

  1. All current slider positions are synced to their HTML attributes before capture
  2. document.documentElement.outerHTML is captured (preserving all current values)
  3. A <style> block is injected at the top of <head> to hide #ui immediately — no rendering flash
  4. The canvas is repositioned as position:fixed filling the full window
  5. A boot script is injected before </body> that overrides resize() to use window.innerWidth / innerHeight, sets AUTO_DISEASE = true, and calls startSim()

The boot script also bakes in the current toggle states so the exported file starts in the same visual configuration:

SHOW_COLOR   = true/false    // colour mode vs greyscale
SPRITE_L     = 50–100        // tint lightness for coloured sprites
SHOW_DETAILS = true/false    // beak / tail / trail / eyes / glow / disease dot

Auto-disease

In the exported file, AUTO_DISEASE = true activates an automatic outbreak timer:

_autoDisTimer -= dt
if _autoDisTimer <= 0:
    _autoDisTimer = random(30, 90)   # seconds
    infect one random healthy boid

Auto-restart (kickstart)

Also gated on AUTO_DISEASE = true, the exported file monitors for a total extinction event. When all boids are gone and all predators are dead, a 3-second countdown begins. If the situation persists, startSim() is called automatically — the simulation restarts from scratch so the display never goes dark:

if boids.length === 0 and all predators dead:
    _allDeadTimer += dt
    if _allDeadTimer >= 3:
        _allDeadTimer = 0
        startSim()
else:
    _allDeadTimer = 0

The 3-second grace period prevents an instant restart if the last boid dies mid-frame alongside a predator that is also on its last health point.

Performance

The frame loop is capped at 30 fps. dt is computed from actual elapsed time so simulation speed is framerate-independent.

Population Pyramid

A 260×200 canvas below the controls shows a live age-sex pyramid, redrawn every pyramid refresh simulation-seconds. It is part of #ui and is therefore hidden in exported files.

Layout

  • Y axis — age bands of 5 simulation-seconds each; youngest band at the bottom, oldest at the top. The number of bands grows automatically if any boid outlives BASE_LIFESPAN (possible at low population pressure where the Gompertz scale reaches 1.25×).
  • X axis (left) — male count, bars drawn right-to-left from the centre divider; colour rgba(84,160,255,0.8).
  • X axis (right) — female count, bars drawn left-to-right from the centre divider; colour rgba(253,121,168,0.8).
  • Centre column — 30 px label area showing the band start age (0, 5, 10 … seconds), flanked by two faint vertical divider lines.
  • Headers — ♂ (blue) and ♀ (pink) above their respective halves, age above the centre column.

Scale and clipping

maxCount = max(1, ceil(TARGET_POP / 4))
barWidth = min(count / maxCount × halfWidth, halfWidth)

The reference value TARGET_POP / 4 is fixed for a given target-population setting. A bar reaches full half-width when a single age-sex band contains ¼ of the target population. Bars that would exceed half-width are clipped at the centre divider so they never bleed into the opposite side. The scale updates if the user moves the target-population slider.

Timing

Redrawn when simClock − lastPyramidUpdate ≥ pyramidRefreshInterval. lastPyramidUpdate is reset to −∞ on startSim() so the first frame always draws an initial pyramid. The pyramid is not redrawn while the simulation is paused.
